## 📋 重要澄清（建议纳入定稿）
1. **随机池数量约束**：当前文档中的6个基础商品仅用于表达特惠商店的商品构成示例；正式上线时随机池商品数量以配置为准。若某期有效随机池商品数大于6，则按权重随机抽取6个不重复SKU；若某期配置恰好等于6，则该期刷新结果可固定为全量展示6个SKU。
2. **同批次去重规则**：一次刷新生成6个商品槽位时，默认同一批次内不允许重复刷出同一SKU；若随机池有效商品不足6个，则视为配置错误，不允许发布。
3. **手动刷新影响范围**：手动刷新仅影响“特惠商店随机6商品”的展示快照与对应“每次刷新限购”；不影响每周限购商品、永久限购商品、金币商店每日限购与水晶商店充值档位。
4. **固定商品重置规则**：每周限购商品只在周一0点重置购买状态；永久限购商品永不重置；二者都不因手动刷新而重置。
5. **快照锁定规则**：特惠商店随机商品的价格、产出、货币类型、购买状态均以“本次刷新生成的玩家快照”为准，在下一次刷新前不受后台热更影响；固定展示商品按当前配置读取。
6. **事务原子性**：商店购买必须按“校验资格 → 扣货币/验支付 → 发商品 → 更新限购”做原子提交；若发奖失败，则货币扣减与限购计数都必须回滚，避免出现扣费成功但未到账。
7. **幂等要求**：特惠商店购买与金币兑换都必须支持服务端幂等；同一请求ID重复提交时，只允许成功生效一次。水晶商店的支付幂等、补单、回调重试由IAP领域统一处理，本文件不展开。
8. **物品发放边界**：背包满、货币上限、转邮件补发等问题不由商店模块处理，统一交给物品模块结算；商店模块只负责发起发奖事务并接收结果。
9. **高品质碎片口径**：`SHOP_SKILL_GOLD_WEEK` 中“技能碎片（金）”只是高品质技能碎片的概括说法，不预先绑定某个具体道具碎片；正式上线时应由该期运营配置明确其实际 `itemId`、名称与产出数量。

## 1. 系统概述

### 1.1 功能目的
商店系统是游戏内资源获取的核心入口，为玩家提供金币、水晶、道具的购买渠道，实现IAP+IAA混合变现的商业化目标。

### 1.2 与其他系统关系

| 关联系统 | 关系说明 |
|----------|----------|
| 背包系统 | 商店购买道具后发放到背包 |
| IAP领域 | 水晶商店购买调用IAP领域能力完成支付，发货方式与购买VIP后获得水晶一致 |
| 玩家状态系统 | 限购状态、货币余额管理 |
| 月卡系统 | 商店入口与月卡特权联动 |
| 任务系统 | 商店刷新、限购刷新关联 |

### 1.3 商店类型

| 商店 | 货币 | 内容 |
|------|------|------|
| 特惠商店 | 金币+水晶 | 各类道具、碎片、礼包 |
| 水晶商店 | 人民币 | 水晶充值 |
| 金币商店 | 水晶 | 金币购买 |

### 1.4 设计约束
- **商业化定位**：商店主要是让玩家可以通过付费快速获得资源
- **限购机制**：特惠商店商品采用限购策略，促进付费转化
- **价值锚定**：遵循1RMB=10水晶的价值体系
- **用户体验**：购买流程简洁，支持快速补货

- **自动刷新**：每日0点（北京时间）自动刷新特惠商店随机6商品；若商品为每周限购商品，则仅在周一0点重置购买状态；若商品为永久限购商品，则永不重置
- **手动刷新**：玩家可消耗水晶立即刷新商品列表（每次刷新消耗5水晶）
- **刷新内容**：每日自动刷新或手动刷新时，仅重置特惠商店随机6商品对应的“每次刷新限购”并生成新的6个商品槽位；周限购/永久限购不会被每日刷新或手动刷新影响
- **展示口径**：一次刷新生成的一组商品会持续展示到下一次刷新事件发生；当天重复进入商店不会重新随机

#### 2.4.2 限购状态管理

```
玩家点击购买
        ↓
检查商品是否存在
        ↓
检查货币余额是否充足
        ↓
按商品配置检查限购次数是否已满（每次刷新/每日/每周/永久）
        ↓ [已满]
返回错误：已购买过/本周期限购已满
        ↓ [未满]
扣除货币
        ↓
发放商品到背包
        ↓
更新限购计数
        ↓
返回购买成功
```

## 🧱 测试建模口径（统一按商品配置+玩家商店状态处理）

为满足“随机展示、固定展示、IAP充值、资源兑换统一按商店商品处理”的实现目标，测试用例统一采用“商品配置 + 玩家商店快照 + 玩家限购状态”三层建模。特惠商店的随机商品不是每次进界面实时随机，而是刷新事件发生时生成玩家级快照；周限购与永久限购商品按固定入口展示；水晶商店直接调用IAP领域能力并根据返回结果发货；金币商店按兑换商品直接结算。

### 3.1 数据对象定义

#### 3.1.1 商品配置对象（shopGoodsConfig）

| 字段 | 含义 |
|------|------|
| `goodsId` | 商店商品唯一ID，如 `SHOP_SKILL_GREEN_5` / `CRYSTAL_001` / `GOLD_003` |
| `shopType` | 商店类型：`special` / `crystal` / `gold` |
| `displayMode` | 展示模式：`random`（随机池）、`fix`（固定展示）、`iap`（支付档位）、`exchange`（资源兑换） |
| `costType` | 消耗货币类型：金币/水晶/人民币 |
| `costValue` | 消耗数量 |
| `rewardItems` | 发放内容列表，每项包含 `itemId` 与 `count` |
| `limitType` | 限购类型：`per_refresh` / `daily` / `weekly` / `permanent` / `none` |
| `limitValue` | 对应周期内最大可购买次数 |
| `weight` | 随机权重，仅 `displayMode=random` 时生效 |
| `activeTimeRange` | 生效时间范围，活动商品可配置起止时间 |

说明：
- `rewardItems` 为统一发奖口径，金币、水晶、普通道具、礼包拆包产物都按物品列表结算。
- 周限购、永久限购商品也属于 `shopGoodsConfig`，只是 `displayMode=fix`，不进入随机池。
- 若 `displayMode=random` 的有效商品数大于6，则需按权重抽取6个不重复SKU；若有效商品数等于6，则允许直接展示全量6个SKU。

#### 3.1.2 玩家商店快照对象（shopSnapshot）

| 字段 | 含义 |
|------|------|
| `snapshotId` | 本次刷新生成的快照ID |
| `shopType` | 商店类型，当前仅特惠商店需要快照 |
| `slotEntries` | 当前6个槽位对应的商品实例列表；每个槽位至少包含 `slotIndex`、`goodsId`、`resolvedRewardItems`、`resolvedCostType`、`resolvedCostValue` |
| `generatedAt` | 本次快照生成时间 |
| `expireAt` | 本次快照失效时间；默认下一个自动刷新时点 |
| `refreshSource` | 刷新来源：`daily_reset` 或 `manual_refresh` |

说明：
- 玩家当天重复进入特惠商店时，只读取同一个有效 `snapshotId`。
- 手动刷新本质是立即废弃旧快照并生成新快照。
- 若某商品产出数量、售价或折扣在每次刷新时可浮动，则以 `slotEntries` 中实际生成的结果为准，而不是实时回读配置值。
- `snapshotId` 是“每次刷新限购”判定的核心维度。

#### 3.1.3 玩家限购状态对象（shopLimitState）

| 字段 | 含义 |
|------|------|
| `goodsId` | 商品ID |
| `limitType` | 限购类型 |
| `progress` | 当前周期已购买次数 |
| `snapshotId` | 当 `limitType=per_refresh` 时关联的快照ID |
| `cycleKey` | 当 `limitType=daily/weekly/permanent` 时对应周期标识 |
| `lastBuyAt` | 最近一次成功购买时间 |

说明：
- `per_refresh` 限购按 `账号 + snapshotId + goodsId` 维度记录。
- `daily` 限购按 `账号 + dateKey + goodsId` 维度记录。
- `weekly` 限购按 `账号 + weekKey + goodsId` 维度记录。
- `permanent` 限购按 `账号 + goodsId` 维度永久保留。

### 3.2 统一判定口径

| 规则项 | 统一口径 |
|--------|----------|
| 特惠商店展示 | 读取玩家当前有效 `shopSnapshot`；无有效快照时才生成新快照；前端展示以 `slotEntries` 中的实际槽位数据为准 |
| 随机商品生成 | 当有效随机池商品数大于6时，从 `displayMode=random` 的商品池中按权重抽取6个不重复SKU；当有效商品数等于6时，直接展示该6个SKU |
| 同日重复进入 | 当天多次进入特惠商店，若未发生自动/手动刷新，则展示同一批商品 |
| 自动刷新 | 每日北京时间00:00为所有玩家重置特惠商店快照，并重置旧快照上的“每次刷新限购” |
| 手动刷新 | 先校验水晶≥5，再扣除5水晶并立即生成新 `shopSnapshot`；成功后旧快照失效 |
| 手动刷新影响范围 | 仅影响特惠商店随机商品快照及其 `per_refresh` 限购状态 |
| 每次刷新限购 | 同一 `snapshotId` 内每个商品最多购买配置次数；新快照生成后重新计算 |
| 每日限购 | 以北京时间自然日为准；金币商店每日限购商品到00:00恢复可购买 |
| 每周限购 | 以北京时间周一00:00为准；每周限购商品到重置点恢复可购买 |
| 永久限购 | 成功购买后永久不可再次购买；可隐藏或展示为永久售罄 |
| 固定商品展示 | 周限购、永久限购商品固定展示，不参与随机池，不占用随机6槽位 |
| 商品购买资格 | 先校验商品存在、商品当前可见、周期内未超限、货币足够，再允许购买 |
| 购买结算顺序 | 特惠商店与金币商店在资格校验通过后，执行扣货币、发商品、记限购，三步必须同事务提交；水晶商店购买能力直接复用IAP领域方法，本文件不定义其支付链路细节 |
| 发奖失败处理 | 任一步失败都视为整笔购买失败，货币扣减与限购计数不得生效 |
| 水晶充值发货 | 水晶商店支付统一走IAP领域；商店侧仅负责调用IAP领域充值方法；发货方式与购买VIP后获得水晶一致；是否支付成功、是否补单、是否重试均以IAP领域返回结果为准，本文件不展开其内部机制 |
| 金币兑换结算 | 金币商店不走IAP支付链路，成功扣除水晶后立即发放金币并更新每日限购 |
| 配置热更影响 | 随机商品快照一经生成，槽位中的商品ID、实际数量、价格与货币类型都锁定到下一次刷新；固定商品按当前配置实时读取 |
| 物品上限处理 | 商店模块不处理背包满、货币溢出、邮件补发，由物品模块统一处理 |
| 时间基准 | 所有刷新、限购重置、活动生效时间均以服务器北京时间为准 |

## ✅ 核心场景用例（仅逻辑与预期数据）
