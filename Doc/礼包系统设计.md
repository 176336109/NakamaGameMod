## 1. 系统概述

### 1.1 功能目的
礼包系统是游戏内IAP付费的核心载体，通过不同类型的礼包满足玩家的差异化付费需求，包括日常付费、活动参与和首次充值引导。

### 1.2 与其他系统关系
- **背包系统**：礼包奖励统一发放至背包，包含货币、碎片、技能、舰船等内容
- **订单系统**：支付完成后触发礼包发放
- **活动系统**：活动礼包与活动周期关联
- **首充系统**：首充状态判断与首充礼包关联

### 1.3 设计约束
- 每日礼包：每天每档限制1次
- 活动礼包：活动期间每档限制1次
- 首充礼包：两档各自永久限购1次，可分别购买1次

## 2. 核心需求

### 2.1 礼包类型

#### 2.1.1 每日礼包
- **刷新周期**：每日0点（北京时间）重置
- **限制逻辑**：每种礼包每天只能购买1次
- **礼包档位**：
  - 特惠礼包（低价引流）
  - 中档礼包（月卡补充）
  - 高档礼包（性价比之选）
- **正式配置**：
  - 6元礼包：水晶×80 + 金币×15,000（1.33x性价比）
  - 30元礼包：水晶×400 + 金币×50,000
  - 98元礼包：水晶×1,500 + 金币×200,000

#### 2.1.2 活动礼包
- **刷新周期**：跟随活动周期，活动结束后重置
- **限制逻辑**：每种礼包在活动期间只能购买1次
- **奖励发放**：礼包中配置的货币、碎片、技能、舰船统一按奖励ID发放至背包；礼包侧不解释奖励ID含义，统一由背包侧按对应ID处理
- **礼包档位**：
  - 活动特惠包
  - 活动豪华包
  - 活动终极包
- **正式配置**：
  - 68元礼包：水晶×800 + 时光沙漏×10 + 金币×50,000
  - 198元礼包：水晶×3,000 + 时光沙漏×30 + 改装件碎片（紫）×10
  - 328元礼包：水晶×6,000 + 时光沙漏×60 + 改装件碎片（金）×5 或 随机舰船

#### 2.1.3 首充礼包
- **刷新周期**：永久限购，不重置
- **限制逻辑**：首充6元与首充30元为两个独立档位，各自永久限购1次；玩家可分别购买1次，购买过对应档位后不可再次购买该档位
- **触发条件**：玩家首次充值后解锁首充礼包入口，两个档位按各自购买状态独立判定
- **奖励发放**：首充奖励中的货币、碎片、技能、舰船统一按奖励ID发放至背包；礼包侧不解释奖励ID含义，统一由背包侧按对应ID处理
- **分天奖励规则**：首充30元礼包采用“购买当天记为第1天”的3天奖励结构；购买当天可立即领取第1天奖励，跨过次日北京时间0点后可领取第2天奖励，再跨过1个北京时间0点后可领取第3天奖励
- **补领规则**：已到可领取时间但尚未领取的奖励不会过期，玩家未领取时可一直保留可领取状态，直到对应天数奖励被领取
- **正式配置**：
  - 首充6元：水晶×120（2倍）+ 新手舰船×1
  - 首充30元：水晶×400 + 改装件碎片（绿）×30 + 技能碎片×15（第1天）+ 改装件碎片（蓝）×15 + 技能碎片×12（第2天）+ 紫色技能×1（第3天）

## 🧱 测试建模口径（统一按礼包配置+玩家礼包状态处理）

为满足“日礼包、活动礼包、首充礼包统一按礼包配置购买，且首充30元支持跨天领取”的实现目标，测试与实现统一采用“礼包配置 + 玩家礼包购买状态 + 分天奖励领取状态”三层建模。礼包模块负责资格判定、触发支付成功后的发奖与状态更新，不直接展开IAP内部支付链路。

### 3.1 数据对象定义

#### 3.1.1 礼包配置对象（giftPackConfig）

| 字段 | 含义 |
|------|------|
| `packId` | 礼包唯一ID，由正式配置自行定义 |
| `packType` | 礼包类型：`daily`=每日礼包，`activity`=活动礼包，`first_recharge`=首充礼包 |
| `packName` | 礼包名称 |
| `priceCurrency` | 支付货币类型；当前统一为 `rmb` |
| `priceAmount` | 支付金额 |
| `limitType` | 限购类型：`daily`=每日限购，`activity_once`=活动期内限购1次，`permanent`=永久限购 |
| `limitValue` | 当前周期最大可购买次数 |
| `activityId` | 活动ID；仅活动礼包需要 |
| `activeTimeRange` | 生效时间范围；每日礼包与首充礼包可为空 |
| `immediateRewardItems` | 购买成功后立即发放的奖励列表 |
| `firstRecharge30DayRewards` | 首充30元礼包的分天奖励列表；其他礼包为空 |

说明：
- 礼包配置支持配置多个奖励物品，统一按奖励列表结算。
- 每日礼包与活动礼包通常只有 `immediateRewardItems`。
- 首充30元同时包含立即发放奖励与分天领取奖励，因此 `immediateRewardItems` 与 `firstRecharge30DayRewards` 都可能有值。
- 礼包ID只要求全局唯一即可，命名规则由正式配置自行确定，本稿不预设固定值。

#### 3.1.2 奖励项对象（giftRewardItem）

| 字段 | 含义 |
|------|------|
| `itemId` | 奖励物品ID |
| `count` | 数量；若为单个技能/舰船则填 `1` |

说明：
- 礼包中的奖励统一按物品列表配置，可同时包含多个物品。
- 礼包侧只按 `itemId + count` 配置并发奖，不区分货币、碎片、技能、舰船的解释逻辑。
- 舰船等物品的具体ID解释由背包侧按既有物品体系处理，礼包模块只负责按配置ID发放。
- 单次发奖内的所有奖励统一按列表结算，不因货币、碎片、技能、舰船而拆分多次请求。

#### 3.1.3 首充30元分天奖励对象（firstRecharge30DayReward）

| 字段 | 含义 |
|------|------|
| `dayIndex` | 第几天可领取，取值从1开始 |
| `rewardItems` | 当天可领取的奖励列表 |

说明：
- 该对象仅用于首充30元礼包的3天奖励配置。
- 首充30元固定为3天奖励，不扩展到其他礼包类型。
- 第1天奖励允许在购买成功后立即领取。

#### 3.1.4 玩家礼包购买状态对象（giftPackPurchaseState）

| 字段 | 含义 |
|------|------|
| `packId` | 礼包ID |
| `packType` | 礼包类型 |
| `limitType` | 限购类型 |
| `progress` | 当前周期已购买次数 |
| `cycleKey` | 对应限购周期标识；每日礼包使用 `dateKey`，活动礼包使用 `activityId`，永久礼包可为空 |
| `lastBuyAt` | 最近一次成功购买时间 |
| `lastOrderId` | 最近一次成功购买的订单ID |

说明：
- 每日礼包按 `账号 + dateKey + packId` 维度记录购买状态。
- 活动礼包按 `账号 + activityId + packId` 维度记录购买状态。
- 首充礼包按 `账号 + packId` 维度永久记录购买状态。
- `lastOrderId` 用于支付回调幂等校验。

#### 3.1.5 首充分天奖励状态对象（firstRechargeStageState）

| 字段 | 含义 |
|------|------|
| `packId` | 对应首充礼包ID |
| `purchaseAt` | 购买成功时间 |
| `dayStates` | 分天奖励状态列表 |

说明：
- 只有存在分天奖励的礼包才需要该对象。
- 购买成功后立即创建该状态对象，并写入全部天数的领取计划。

#### 3.1.6 首充分天槽位状态对象（firstRechargeDayState）

| 字段 | 含义 |
|------|------|
| `dayIndex` | 第几天奖励 |
| `unlockAt` | 该天奖励变为可领取的服务器时间 |
| `status` | `locked` / `claimable` / `claimed` |
| `claimedAt` | 实际领取时间；未领取时为空 |

说明：
- 第1天的 `unlockAt = purchaseAt`。
- 第2天的 `unlockAt` 为购买后跨过的第1个北京时间00:00。
- 第3天的 `unlockAt` 为购买后跨过的第2个北京时间00:00。
- 奖励一旦进入 `claimable`，即使继续跨天也不会失效，直到被领取。

### 3.2 统一判定口径

| 规则项 | 统一口径 |
|--------|----------|
| 礼包展示 | 前端展示以当前有效 `giftPackConfig` 为准；每日礼包、活动礼包、首充礼包按各自条件独立显示 |
| 首充入口显示 | 玩家完成首次充值后显示首充礼包入口；未完成首次充值前不可购买首充礼包 |
| 每日礼包限购 | 以北京时间自然日为准；同一 `packId` 每日最多购买配置次数，00:00后重置 |
| 活动礼包限购 | 以 `activityId` 为准；同一活动周期内每个礼包最多购买配置次数，活动结束后重置 |
| 首充礼包限购 | 首充6元与首充30元按两个独立 `packId` 永久限购；每档各自最多购买1次 |
| 商品购买资格 | 先校验礼包存在、当前时间在有效期内、未超过限购，再允许发起购买；首充礼包额外校验玩家已完成首次充值 |
| 支付成功处理 | 礼包模块仅在收到订单系统确认成功后才发奖并更新购买状态 |
| 支付幂等 | 同一订单ID重复回调时，只允许一次真正生效；重复请求返回幂等成功，不重复发奖、不重复记限购 |
| 立即奖励发放 | `immediateRewardItems` 在购买成功后立即按同一事务发放 |
| 分天奖励建档 | 购买首充30元礼包成功后，需同步创建对应的分天奖励状态对象 |
| 首充30元第1天规则 | 购买成功当天立即解锁第1天奖励，可在购买成功后立刻领取 |
| 首充30元跨天规则 | 购买后每跨过1个北京时间00:00，依次解锁下一天奖励；购买当天始终记为第1天 |
| 首充30元补领规则 | 已解锁但未领取的天数奖励永久保留为可领取状态，不因继续跨天而失效 |
| 首充30元领取资格 | 仅当对应 `dayState.status=claimable` 时允许领取；`locked` 与 `claimed` 状态不可重复领取 |
| 购买与领取事务 | 购买按“校验资格 → 等待订单成功 → 发立即奖励 → 更新购买状态/创建分天状态”提交；分天奖励领取按“校验资格 → 发奖励 → 更新领取状态”提交 |
| 发奖失败回滚 | 购买或领取过程中若发奖失败，则本次操作整体失败；购买状态、分天状态、领取状态不得部分生效 |
| 配置热更影响 | 礼包价格、限购、奖励内容在购买时按当前配置结算；已创建的分天奖励状态按购买时写入的奖励计划执行，不回读新配置 |
| 物品ID口径 | 礼包奖励统一按配置中的 `itemId` 发放；礼包模块不解释物品类型含义，由背包侧按对应ID处理 |
| 时间基准 | 每日重置、活动生效、首充30元分天解锁均以服务器北京时间为准 |

| B06 | 首充6元已购后不可再次购买 | 某6元首充礼包已有永久购买记录 | 再次购买6元首充礼包 | 返回“该档位已购买”；不重复发奖 |
