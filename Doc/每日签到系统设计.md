## 1. 系统概述

### 1.1 功能目的
每日签到系统是游戏内重要的免费资源获取渠道，通过7天循环签到机制，为玩家提供持续登录动力，同时通过补签功能提供付费补救途径。

### 1.2 与其他系统关系
- **物品系统**：金币、水晶、时光沙漏、碎片等奖励统一按普通物品堆叠实例发放、扣减与消耗
- **背包系统**：签到奖励与补签消耗都通过统一物品体系结算
- **时间系统**：签到状态重置以服务器时间为准

### 1.3 设计约束
- 签到周期固定为7天
- 补签消耗固定为20水晶/次
- 跨天以服务器时间（北京时间UTC+8）为准
- 签到周期按服务器自然日推进，不因玩家是否漏签而停留或延长
- 同一自然日每个账号只能对“今日槽位”签到1次

## 2. 核心需求

### 2.1 签到周期机制
- **周期长度**：7天循环
- **周期起算**：玩家账号创建所在自然日记为签到体系的起算日，第1个签到周期第1天从该自然日开始
- **状态创建**：玩家首次进入签到界面或首次发起签到相关请求时，按账号创建时间推导当前所在周期，并创建对应的当前周期状态对象
- **周期推进**：签到周期按服务器北京时间自然日连续推进7天；每过1个自然日推进1个周期日槽位，不因玩家是否登录而暂停
- **周期重置**：当前周期开始后第8个自然日到来时，无论玩家是否漏签，均进入下一个新周期
- **进度保留**：断签不重置当前7天周期内已签到/已补签记录，但不会阻止周期自然推进
- **跨周期清理**：进入新周期后，上一周期未补签天数全部作废，上一周期补签入口关闭

### 2.2 签到奖励配置
7天签到奖励表（正式配置）：

| 天数 | 奖励类型 | 奖励ID | 奖励数量 | 品质 |
|------|----------|--------|----------|------|
| 第1天 | 金币 | 1 | 100 | - |
| 第2天 | 水晶 | 2 | 50 | - |
| 第3天 | 时光沙漏 | 010300001 | 1 | - |
| 第4天 | 技能碎片 | 020100001 | 5 | 绿 |
| 第5天 | 改装件碎片 | 030100001 | 3 | 绿 |
| 第6天 | 金币、水晶 | 1、2 | 200、30 | - |
| 第7天 | 时光沙漏 | 010300001 | 2 | - |

> 📊 奖励价值：第7天2个时光沙漏 = 10水晶价值，总周期价值约135水晶（不含金币）

### 2.3 签到与补签规则
- **正常签到触发条件**：玩家当天对应的周期槽位尚未被签到或补签
- **正常签到限制**：仅可签到“今日槽位”，不可提前签到未来槽位
- **补签触发条件**：当前周期内存在历史漏签天数，且玩家拥有足够水晶
- **补签消耗资源**：20水晶/次（固定）
- **补签范围**：仅可补签当前周期内、且早于今天槽位的漏签天数；不可补签未来天数，不可跨周期补签
- **补签次数**：当前周期内不限总次数，但每个漏签槽位只能补签1次
- **补签奖励**：按被补签那一天的正式奖励配置发放
- **同日行为**：玩家可在同一自然日内先签到今日，再补签历史漏签；也可先补签后签到今日

### 2.4 签到状态显示
- 已签到：显示勾选标记
- 已补签：显示补签标记，并与普通签到区分
- 今日可签到：显示可领取状态
- 历史漏签可补：显示补签入口与补签消耗
- 未来天数：显示未解锁状态，不可操作

## 🧱 测试建模口径（统一按签到配置+玩家周期状态处理）

为满足“7天自然周期推进 + 当日签到 + 历史补签 + 统一物品结算”的实现目标，测试用例统一采用“签到奖励配置 + 玩家签到周期状态 + 玩家槽位状态”三层建模。签到模块不直接处理物品上限与支付能力，只负责判定资格、触发资源扣减和奖励发放，并记录状态。

### 3.1 数据对象定义

#### 3.1.1 签到奖励配置对象（signRewardConfig）

| 字段 | 含义 |
|------|------|
| `dayIndex` | 周期日序号，取值1-7 |
| `rewardItems` | 当日奖励列表，每项包含 `itemId` 与 `count` |
| `makeupCostItem` | 补签消耗对象，包含 `itemId` 与 `count`；当前固定为 `{ itemId: 2, count: 20 }` |

说明：
- 第6天这类复合奖励统一按 `rewardItems` 多条记录表达，不单独做特殊逻辑。
- 若未来调整补签价格，应只改 `makeupCostItem` 配置，不改状态结构。

#### 3.1.2 玩家签到周期状态对象（signCycleState）

| 字段 | 含义 |
|------|------|
| `cycleId` | 当前签到周期唯一ID |
| `cycleStartDate` | 当前周期开始的服务器自然日；由账号创建时间向下推导到当前所在周期的起始日 |
| `cycleEndDate` | 当前周期结束的服务器自然日 |
| `currentDayIndex` | 基于当前服务器日期推导出的当期天数，范围1-7 |
| `dayStates` | 7个槽位的状态列表 |
| `lastRefreshDate` | 最近一次按自然日刷新周期状态的日期 |

说明：
- `currentDayIndex` 可运行时按服务器日期与 `cycleStartDate` 推导，也可冗余存储，最终以服务器计算结果为准。
- 玩家首次触发签到系统时，若账号已创建多日，则需先按账号创建时间推导当前周期的 `cycleStartDate` 与 `currentDayIndex`，再创建当前周期状态。
- 第8个自然日到来时，旧 `cycleId` 失效，创建新的 `cycleId` 与新的7天槽位状态。

#### 3.1.3 槽位状态对象（signDayState）

| 字段 | 含义 |
|------|------|
| `dayIndex` | 周期内第几天 |
| `status` | `unsigned` / `signed` / `makeup_signed` |
| `claimAt` | 成功领取奖励时间 |
| `claimType` | 领奖类型：`normal` 表示正常签到领取，`makeup` 表示补签领取 |

说明：
- `unsigned` 仅表示该槽位尚未领取，不区分“今天可签”与“历史漏签”，这两个前端展示状态由 `dayIndex` 与 `currentDayIndex` 运行时判定。
- `claimType=normal` 时，表示该槽位是玩家在对应自然日完成正常签到后领取的奖励。
- `claimType=makeup` 时，表示该槽位是玩家在后续自然日通过补签方式补领的奖励。
- 每个 `dayIndex` 最多只会成功进入一次已领取状态，不能重复签到或重复补签。

### 3.2 统一判定口径

| 规则项 | 统一口径 |
|--------|----------|
| 首次建周期 | 玩家首次进入签到界面或首次发起签到相关请求时，按账号创建时间推导当前所在周期，并创建对应的 `signCycleState` |
| 当前周期推导 | 先取 `accountCreateDateKey=账号创建所在服务器自然日`，再计算“当前服务器自然日”与 `accountCreateDateKey` 之间隔了多少个自然日，这个值记为“自然日差值”；`cycleOffset = floor(自然日差值 / 7)` 的含义是：每满7个自然日，就说明已经完整跨过了1个签到周期，`cycleOffset` 就加1；不足7天的余数仍属于当前这一个周期。最后再用 `cycleStartDate = accountCreateDateKey + cycleOffset * 7天` 得到当前所在周期的起始日 |
| 今日槽位计算 | `currentDayIndex = 当前服务器自然日 - cycleStartDate + 1`，取值范围1-7 |
| 周期切换 | 当进入下一个自然日后 `currentDayIndex` 应大于7时，旧周期结束并创建新周期，新周期第1天对应新的 `cycleStartDate` |
| 今日签到资格 | 仅当今日槽位状态为 `unsigned` 时，允许正常签到 |
| 历史补签资格 | 仅当 `dayIndex < currentDayIndex` 且槽位状态为 `unsigned` 时，允许补签 |
| 未来天数限制 | `dayIndex > currentDayIndex` 的槽位不可签到、不可补签 |
| 同日签到与补签 | 正常签到与历史补签互不冲突，可在同一天内同时发生 |
| 补签扣费 | 补签前先校验水晶余额是否足够；不足则补签失败 |
| 结算顺序 | 正常签到按“校验资格 → 发奖励 → 更新状态”提交；补签按“校验资格 → 扣20水晶 → 发奖励 → 更新状态”提交 |
| 发奖失败回滚 | 若奖励发放失败，则本次签到/补签视为失败；补签时已扣除的20水晶也必须回滚 |
| 周期内漏签保留 | 当前周期内漏签槽位会一直保留为可补签状态，直到周期结束或玩家完成补签 |
| 跨周期失效 | 旧周期未领取槽位在新周期开始后全部失效，不补发、不转移 |
| 奖励配置来源 | 每个槽位发什么奖励，以 `signRewardConfig.dayIndex` 对应配置为准 |
| 配置热更影响 | 已进入某个周期但尚未领取的槽位，领取时读取当前配置 |
| 物品上限处理 | 签到模块不处理物品溢出、转邮件补发，由物品模块统一处理 |
| 时间基准 | 所有跨天、周期切换、补签范围判断均以服务器北京时间为准 |

## ✅ 核心场景用例（仅逻辑与预期数据）

| 用例ID | 场景 | 前置数据 | 触发 | 预期数据 |
|--------|------|----------|------|----------|
| C01 | 新账号首次触发创建签到周期 | 玩家当天刚创建账号，且不存在 `signCycleState` | 首次进入签到界面或首次发起签到相关请求 | 创建 `cycleId=C1`；`cycleStartDate=账号创建当日`；`currentDayIndex=1`；初始化7个 `dayStates` 为 `unsigned` |
| C02 | 第1天正常签到成功 | `cycleId=C1`；第1天槽位为 `unsigned` | 点击签到 | 发放第1天奖励金币100；第1天槽位改为 `signed`；记录 `claimType=normal` |
| C03 | 连续两天正常签到 | 第1天已签到；次日到来；第2天槽位为 `unsigned` | 次日点击签到 | 发放第2天奖励水晶50；第2天槽位改为 `signed` |
| C04 | 漏签后次日补签成功 | 第1天未签；当前为第2天；玩家水晶充足 | 对第1天执行补签 | 扣除水晶20；发放第1天奖励金币100；第1天槽位改为 `makeup_signed` |
| C05 | 同日先签到今日再补签历史 | 当前为第3天；第2天漏签；第3天未签；玩家水晶充足 | 先签到第3天，再补签第2天 | 第3天变为 `signed`；第2天变为 `makeup_signed`；补签额外扣20水晶 |
| C06 | 同日先补签历史再签到今日 | 当前为第3天；第2天漏签；第3天未签；玩家水晶充足 | 先补签第2天，再签到第3天 | 两次操作均成功；状态与奖励分别正确结算 |
| C07 | 第6天复合奖励发放 | 当前为第6天；第6天未签 | 点击签到 | 一次性发放金币200与水晶30；第6天槽位改为 `signed` |
| C08 | 同一槽位不可重复领取 | 第4天槽位已为 `signed` | 再次点击第4天签到 | 返回“已领取”；不重复发奖 |
| C09 | 周期自然推进到第7天 | 当前周期第6天已存在状态记录 | 跨到第7天 | `currentDayIndex=7`；前6天状态保留；第7天槽位可按规则签到 |
| C10 | 第8天进入新周期 | 旧周期 `cycleId=C1` 已到第7天结束 | 跨到第8个自然日 | 旧周期失效；创建新 `cycleId=C2`；新周期第1天槽位为 `unsigned` |
| C11 | 补签发放对应历史奖励 | 当前为第7天；第3天漏签；玩家水晶充足 | 补签第3天 | 扣除水晶20；发放第3天时光沙漏1；第3天槽位改为 `makeup_signed` |
| C12 | 奖励发放交由物品模块处理 | 任意签到/补签成功；背包或货币接近上限 | 结算奖励 | 签到模块只发起发奖；最终到账、截断或转邮件按物品模块规则处理 |
