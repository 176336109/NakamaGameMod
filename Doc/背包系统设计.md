# 背包系统设计

**版本**：1.1  
**日期**：2026年3月11日  
**负责人**：马尔斯（主策划）  

## 🎯 设计目标
1. **统一物品存储**：为所有系统提供统一的加物、扣物、用物、过期处理能力。
2. **支撑独立属性定义**：明确可堆叠/不可堆叠，以及永久/有时限是两组独立属性。
3. **降低业务重复实现**：商店、月卡、礼包、任务等系统不再各自维护物品库存逻辑。
4. **保证发放与消耗一致性**：支持幂等发奖、并发使用、容量判断、过期清理等关键场景。

## 🎒 背包概览

### 背包能力范围
| 项目 | 规格 |
|------|------|
| **核心职责** | 统一管理物品增加、扣减、使用、堆叠、占格、过期与查询 |
| **支持对象** | 可堆叠物品、不可堆叠物品；永久物品、有时限物品 |
| **默认容量** | 100格 |
| **容量作用范围** | 仅对需要新增占格对象的入包生效；已有堆叠物品加数量不额外占格 |

### 与其他系统关系
| 关联领域 | 关系说明 |
|----------|----------|
| 商店系统 | 商店购买成功后，请求背包系统发放已解析的物品奖励 |
| 礼包系统 | 礼包系统将 `resolvedRewards` 中的物品部分交给背包系统入包 |
| 月卡系统 | VIP/SVIP权益物品、钻石、时光沙漏等奖励通过背包系统存储与发放 |
| 任务系统 | 任务完成后请求背包系统发奖 |
| 战斗系统 | 战斗掉落资源和物品统一进入背包 |
| 舰船/技能等外部领域 | 非通用物品奖励不进入背包，由对应领域自行解释和发放 |

## 📦 物品属性说明

### 属性维度

| 属性 | 可选值 | 含义 |
|------|--------|------|
| 是否堆叠 | 可堆叠 / 不可堆叠 | 决定是按数量聚合存储，还是按单个实例存储 |
| 是否有时限 | 永久 / 有时限 | 决定是否需要校验 `expireAt` |

说明：
- 本文档只定义“物品如何进背包、如何存储、如何离开背包”，不负责解释舰船、技能等非物品奖励。
- 金币 `1`、水晶 `2`、时光沙漏 `010300001` 等通用道具按普通堆叠物品处理。
- VIP/SVIP月卡直接复用 [VIP_SVIP_PLAN.md](VIP_SVIP_PLAN.md) 中已定义的时效权益实例口径。
- 当前不做玩家间流通，因此以下对象默认挂在玩家自己的背包存储下，不单独保存 `playerId`。

## 🧱 测试建模口径（统一按堆叠属性 + 时限属性处理）

为保证背包系统能与月卡、礼包、商店、任务共用一套实现，测试与实现统一采用“可堆叠物品记录 + 不可堆叠物品实例”的建模口径；是否存在时限由 `expireAt` 单独决定。普通资源通过 `count` 直接累计；VIP/SVIP这类时效权益物品继续复用 VIP 文档对象表达。

### 1. 物品定义对象（itemConfig）

| 字段 | 含义 |
|------|------|
| `itemId` | 物品类型ID |
| `itemName` | 物品名称 |
| `itemDesc` | 物品说明 |
| `itemType` | 物品类型，如 `currency` / `material` / `entitlement` |
| `stackable` | 是否可堆叠 |
| `hasExpireAt` | 是否有时效 |
| `maxStackCount` | 单条堆叠记录上限；不可堆叠物品为空 |
| `useRule` | 使用规则：不可使用、直接消耗、跳转外部系统处理 |
| `occupySlot` | 是否占用背包格子 |

说明：
- 通用道具ID表只负责ID口径；背包仍需维护与存储相关的 `itemConfig`。
- `stackable=true` 的物品统一走聚合存储模型。
- `stackable=false` 的物品统一走实例存储模型。
- `hasExpireAt=true` 的物品都要校验有效期，无论其是否可堆叠。

### 2. 可堆叠物品对象（stackItemRecord）

| 字段 | 含义 |
|------|------|
| `itemId` | 物品类型ID |
| `itemType` | 物品类型，冗余存储用于快速过滤与展示 |
| `count` | 当前持有数量 |
| `hasExpireAt` | 是否有时效；永久物品为 `false`，限时物品为 `true` |
| `expireAt` | 该记录的统一过期时间；永久物品为空 |

说明：
- 该对象与VIP文档中的普通堆叠物品记录口径一致。
- 同一背包下同一 `itemId` 在同一过期批次下只保留一条记录；永久物品通常只有一条。
- `itemType` 取自 `itemConfig`，用于背包页签、筛选和快速展示。
- `hasExpireAt=true` 时必须同时存在 `expireAt`；`hasExpireAt=false` 时 `expireAt` 为空。
- 增加已有 `stackItemRecord` 的 `count` 不新增背包格子。

### 3. 不可堆叠物品实例（当前时效权益样例以VIP文档为准）

| 字段 | 含义 |
|------|------|
| `instanceId` | 物品实例唯一ID |
| `itemId` | 物品类型ID |
| `type` | 物品类型，固定为“时效权益” |
| `subType` | 物品子类型，如“月卡” |
| `description` | 物品描述 |
| `stackable` | 是否可堆叠，固定为 `false` |
| `usable` | 是否可使用，固定为 `false` |
| `startAt` | 生效时间 |
| `expireAt` | 过期时间 |
| `benefitPlanId` | 权益配置ID |

说明：
- 本节字段与 [VIP_SVIP_PLAN.md](VIP_SVIP_PLAN.md) 中“月卡权益道具定义”保持一致，不新增、不删改。
- 当前已明确的不可堆叠实例样例是 VIP/SVIP 这类有时限权益物品。
- 若后续存在不可堆叠永久物品，也仍按单个实例存储，只是不校验 `expireAt`。
- 对于同类时效权益续费，优先更新已有实例而不是新建多条并存实例，具体规则以月卡业务规则为准。

## 3.2 统一判定口径

| 规则项 | 统一口径 |
|--------|----------|
| 入包对象 | 只有已定义 `itemId` 的物品才能进入背包；非物品奖励不进入背包 |
| 属性关系 | `stackable` 与 `hasExpireAt` 为两个独立属性，任何存储逻辑都必须分别判断 |
| 可堆叠物品存储 | `stackable=true` 的物品按 `stackItemRecord` 聚合存储，数量字段统一为 `count` |
| 不可堆叠物品存储 | `stackable=false` 的物品按实例独立存储；当前时效权益实例字段复用 VIP 文档 |
| 有时限生效 | `hasExpireAt=true` 的物品以 `expireAt > 当前时间` 视为有效 |
| 背包容量来源 | 背包最大容量由配置读取，不单独维护背包状态对象 |
| 背包满判定 | 仅在需要新增占格记录时校验容量；已有堆叠记录加数量不受容量限制 |
| 批量发奖事务 | 单次发奖请求中的全部物品按原子事务处理；任一物品无法入包则整批失败，不允许半成功 |
| 发奖幂等 | 同一发奖请求重复到达时，只允许一次真正加物 |
| 扣减幂等 | 同一使用/扣减请求重复到达时，只允许一次真正扣物 |
| 物品使用 | 使用前先校验物品存在，再按是否堆叠、是否有时限分别校验数量、实例状态和有效期 |
| 过期处理 | 到期物品由背包系统移出可用库存；外部业务读取时不得再视为有效 |
| 时效续费 | 若业务定义可续费，则更新已有有时限实例的 `expireAt`；具体规则以对应业务为准 |
| 资源发放 | 金币、水晶、时光沙漏等资源按通用道具ID直接发到对应堆叠记录 |
| 上限处理 | 是否允许超上限、截断或转邮件不在本稿展开；背包至少要在执行前给出“可新增占格/不可新增占格”的明确判定 |
| 删除规则 | `count` 扣减到0的堆叠记录可删除；已过期有时限物品不再计入有效库存 |
| 查询口径 | 对外默认返回“当前有效库存” |

## ✅ 核心场景用例（仅逻辑与预期数据）

| 用例ID | 场景 | 前置数据 | 触发 | 预期数据 |
|--------|------|----------|------|----------|
| C01 | 发放新普通物品 | 背包中不存在 `010300001` 记录；按当前背包数据计算后仍未达到配置容量上限 | 发放 `010300001 x3` | 新增 `stackItemRecord(010300001).count=3` |
| C02 | 发放已有普通物品 | 已存在 `stackItemRecord(2).count=100` | 发放 `2 x80` | `count=180` |
| C03 | 发放可堆叠有时限物品 | 背包中不存在某限时活动代币记录；按当前背包数据计算后仍未达到配置容量上限 | 发放限时活动代币 `x10`，`expireAt=T1` | 新增对应 `stackItemRecord.count=10`；记录 `expireAt=T1` |
| C04 | 发放不可堆叠有时限物品 | 不存在有效 `item_vip_active` 实例；按当前背包数据计算后仍未达到配置容量上限 | 月卡购买成功 | 新增1个 VIP 文档定义的时效物品实例；`itemId=item_vip_active` |
| C05 | 使用时光沙漏 | 已存在 `stackItemRecord(010300001).count=5` | 触发一次自动跳广告使用 | `count=4` |
| C06 | 扣减后数量归零 | 已存在 `stackItemRecord(020200001).count=1` | 消耗 `020200001 x1` | 对应记录删除或 `count=0` 后清理；不再占格 |
| C07 | 发放不可堆叠永久物品 | 按当前背包数据计算后仍未达到配置容量上限 | 发放1个不可堆叠普通物品 | 新增1个实例记录；不设置 `expireAt` |
| C08 | 有时限物品过期清理 | 存在 `item_vip_active.expireAt<Tnow` 或某限时活动代币 `expireAt<Tnow` | 登录或定时清理 | 对应到期物品移出有效库存；查询背包时不再返回这些物品 |
| C09 | 时效权益续费 | 已存在有效 `item_vip_active.expireAt=T1` | 再次购买VIP | 复用原时效物品实例并更新 `expireAt>T1`；不新增第二个有效VIP实例 |
| C10 | 查询有效背包 | 背包中同时存在永久物品、有效限时物品、已过期物品 | 打开背包 | 返回永久物品和当前有效物品；已过期物品不在可用列表中 |
| C11 | 奖励批量入包 | 商店或礼包发奖包含 `2 x400`、`010300001 x10`、`item_vip_active x1`；按当前背包数据计算后容量足够 | 批量发奖 | 可堆叠物品按聚合模型入包，不可堆叠时效物品按 VIP 实例模型入包；整体成功 |
| C12 | 重复发奖请求幂等 | 同一发奖请求已完成发放 | 再次收到同一发奖请求 | 不重复加物；返回幂等成功结果 |

## ⚠️ 边界情况用例（仅逻辑与预期数据）

| 用例ID | 场景 | 前置数据 | 触发 | 预期数据 |
|--------|------|----------|------|----------|
| B01 | 背包满但发放已有堆叠物品 | 按当前背包数据计算后已达到配置容量上限；已存在 `stackItemRecord(2)` | 发放 `2 x100` | 允许成功；仅增加 `count`；不因背包满失败 |
| B02 | 背包满且需新增堆叠记录 | 按当前背包数据计算后已达到配置容量上限；不存在 `010300001` 记录 | 发放 `010300001 x1` | 发放失败；不新增记录 |
| B03 | 背包满且需新增不可堆叠物品 | 按当前背包数据计算后已达到配置容量上限 | 发放 `item_vip_active x1` 或其他不可堆叠物品 | 发放失败；不新增实例 |
| B04 | 同一普通物品并发扣减 | `stackItemRecord(010300001).count=1` | 两个请求同时消耗 `x1` | 最终仅1次成功；`count` 不得为负数 |
| B05 | 可堆叠限时物品跨批次共存 | 同一 `itemId` 同时存在 `expireAt=T1` 与 `expireAt=T2` 两批记录 | 发放或清理其中一批 | 两批记录独立处理；不得因清理 T1 误删 T2 |
| B07 | 奖励一半可堆叠一半需新占格 | 按当前背包数据计算只剩1格；已存在 `2`，不存在 `010300001`、`item_vip_active` | 批量发放 `2 x100` + `010300001 x1` + `item_vip_active x1` | 按批量发奖原子事务处理；因容量不足导致整批失败；三种物品都不落库 |
| B08 | 有时限物品刚过期时被读取 | 某物品 `expireAt` 刚好等于当前时间 | 读取背包状态 | 按“`expireAt > 当前时间` 才有效”判定；此时应视为无效 |
| B09 | 续费请求重复到达 | 已存在VIP时效物品；同一订单重复回调 | 两次续费更新 | 只延长一次有效期，不得多加30天 |
| B10 | 删除数量为0的普通物品 | `stackItemRecord(020100001).count=0` | 读取背包或执行清理 | 该记录应被移除，不继续占格 |
| B11 | 外部奖励误传非物品引用 | 奖励内容为 `rewardRef=random_ship_pool` | 请求入包 | 背包系统拒绝入包，返回“非物品奖励”错误 |
| B12 | 配置名和通用ID不一致 | 外部系统尝试发 `时钟沙漏` | 请求入包 | 应按统一 `itemId=010300001` 校验；不接受错误命名直接入库 |
| B13 | 双端同时查看背包并发生结算 | 双端同时查看同一背包 | 一端使用物品、一端读取背包 | 读取结果应与最新成功结算后的背包数据一致，避免展示旧库存长期覆盖新库存 |
| B14 | 已过期有时限物品仍被业务缓存 | 某有时限物品已过期 | 外部业务继续读旧缓存 | 背包查询接口应返回无效状态；业务不得仅依赖本地缓存长期判定 |

## 附录：与VIP月卡设计的对齐点

| 对齐项 | 背包统一口径 |
|--------|--------------|
| 普通物品 | 继续使用 VIP 文档中的 `count` 堆叠记录思路 |
| 是否堆叠 | 作为独立存储属性判断，不和时限属性绑定 |
| 是否有时限 | 作为独立有效期属性判断，不和堆叠属性绑定 |
| 时效权益实例 | 直接复用 VIP 文档中的权益实例定义，不新增字段 |
| VIP权益物品 | `item_vip_active` 作为时效实例物品入包 |
| SVIP权益物品 | `item_svip_active` 作为时效实例物品入包 |
| 钻石 | 作为普通堆叠物品处理 |
| 时光沙漏 | 使用通用道具ID `010300001`，作为普通堆叠物品处理 |
